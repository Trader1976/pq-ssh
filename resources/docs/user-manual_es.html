Qué es PQ-SSH

CPUNK PQ-SSH es un cliente SSH basado en perfiles, centrado en un flujo de trabajo de terminal limpio, valores predeterminados de seguridad sólidos y un indicador de preparación post-cuántica (PQ).

Primero la terminal Perfiles Macros SFTP Registro de auditoría
Primero la terminal

Las conexiones abren una terminal SSH real impulsada por tu OpenSSH del sistema.

Perfiles

Guarda host/usuario/puerto y preferencias. Conecta con un solo clic.

Macros

Asigna teclas rápidas a comandos y envíalos al terminal SSH activo al instante.

Flujos de trabajo SFTP

Una sesión opcional de libssh permite navegar y transferir archivos sin romper la experiencia de terminal.

Primeros pasos

  1. Abre PQ-SSH.
  2. Haz clic en Editar perfiles….
  3. Crea un perfil (Nombre, Usuario, Host/IP, Puerto).
  4. Selecciona un perfil y haz clic en Conectar (o haz doble clic en el perfil).
Comportamiento predeterminado: “Abrir nueva conexión en NUEVA ventana” está habilitado, por lo que cada conexión se abre por separado.

Perfiles

Un perfil almacena los detalles de conexión y las preferencias de la interfaz:

  • Nombre — etiqueta amigable (p. ej. “Servidor#1”)
  • Usuario — nombre de usuario SSH (p. ej. root)
  • Host — nombre de host o IP (p. ej. 192.168.0.1)
  • Puerto — el valor predeterminado es 22
  • Grupo — etiqueta opcional para organizar perfiles en la lista
  • Tipo de clave — normalmente auto o openssh
  • Esquema de terminal — esquema de colores para la terminal
  • Macros de teclas rápidas — lista de macros por perfil (atajo → comando)
Consejo: Mantén los nombres de perfil cortos y consistentes (Prod, Staging, Home, etc.) y usa Grupos para mantener la lista legible.
Administrador de perfiles — crea y organiza perfiles (con opciones avanzadas).

Reenvío de puertos

PQ-SSH admite reenvío de puertos por perfil. Esto te permite “tunelizar” tráfico TCP de forma segura a través de tu conexión SSH. Puedes habilitar múltiples reglas de reenvío en un solo perfil (multi-forward).

Dónde configurarlo

  1. Abre Editar perfiles…
  2. Selecciona un perfil
  3. En la columna central, abre Avanzado
  4. Habilita Habilitar reenvío de puertos
  5. Haz clic en Reenvío de puertos… para añadir/editar reglas

Tipos de reenvío

  • Reenvío local (-L) — reenvía un puerto local de tu ordenador a un host/puerto de destino en el lado remoto.
    Ejemplo: acceder a un servicio web remoto localmente: 
    ssh -L 127.0.0.1:18888:localhost:18080 user@server
    Luego navegar/curl: 
    curl http://127.0.0.1:18888/
  • Reenvío remoto (-R) — reenvía un puerto en el servidor remoto hacia un host/puerto accesible desde tu lado local.
    Ejemplo (conceptual): 
    ssh -R 0.0.0.0:9000:127.0.0.1:3000 user@server
    Los clientes remotos pueden conectarse al puerto 9000 del servidor remoto (según la política del servidor).
  • Reenvío dinámico (-D) — crea un proxy SOCKS local.
    Ejemplo: 
    ssh -D 127.0.0.1:1080 user@server
    Luego puedes apuntar un navegador o herramienta a SOCKS5 127.0.0.1:1080.

Campos de la regla (qué significan)

  • Habilitado — activa/desactiva una regla individual (sin borrarla)
  • Tipo — Local (-L), Remoto (-R), Dinámico (-D)
  • Dirección de enlace — dónde se crea el listener (predeterminado: 127.0.0.1 para solo local)
  • Puerto de escucha — el puerto a abrir (1–65535)
  • Host de destino / Puerto de destino — solo para reglas Local/Remoto (se ignora en Dinámico)
  • Descripción — nota opcional para recordar para qué es la regla
Consejo de seguridad: Prefiere enlazar a 127.0.0.1 para reenvíos locales. Enlazar a 0.0.0.0 puede exponer el puerto reenviado a toda tu red.

Cómo probar el reenvío (lista rápida)

  1. Inicia un servicio en el host remoto (ejemplo: servidor de prueba Python):
    python3 -m http.server 18080 --bind 127.0.0.1
  2. En PQ-SSH, añade un reenvío Local:
    127.0.0.1:18888 → localhost:18080
  3. Conecta con el perfil.
  4. En tu máquina local, pruébalo:
    curl -I http://127.0.0.1:18888/

Solución de problemas del reenvío

  • “Address already in use” significa que el puerto de escucha ya está ocupado. Elige otro puerto (p. ej. 18890).
  • Connection reset / timeout normalmente significa que el servicio de destino no está ejecutándose o no es accesible desde el lado remoto.
  • El reenvío remoto (-R) no funciona puede estar bloqueado por la política del servidor (AllowTcpForwarding, GatewayPorts).
Los reenvíos de PQ-SSH se implementan usando tu OpenSSH del sistema pasando opciones -L/-R/-D al lanzar ssh. No se aplican ajustes de reenvío al servidor automáticamente.

Sondear criptografía del servidor (diagnóstico)

En Editar perfiles…, PQ-SSH ofrece un botón Sondear criptografía del servidor… para ayudarte a inspeccionar qué negocia el handshake SSH con un servidor.

  • Cuándo está disponible: se habilita después de configurar Usuario y Host (Puerto es opcional).
  • Qué muestra: KEX, algoritmo de clave de host y cifrados negociados.
  • Cómo funciona: ejecuta tu cliente OpenSSH del sistema en modo verboso (ssh -vvv) con un tiempo de espera corto.
  • No requiere autenticación: la negociación ocurre antes de autenticarse.
  • Solo lectura: no modifica el servidor ni el perfil.
Si el sondeo muestra (not found), causas comunes incluyen: host inaccesible, puerto bloqueado, fallo de DNS o desajuste al parsear la salida. Abre los detalles del diálogo de sondeo para ver la salida sin procesar.
Consejo: Usa el sondeo para confirmar algoritmos modernos (p. ej. curve25519-sha256, ssh-ed25519) y detectar despliegues heredados.

Conectar

  1. Selecciona un perfil de la lista.
  2. Haz clic en Conectar (o haz doble clic en el perfil).
  3. Se abre una ventana de terminal y SSH inicia automáticamente.
Las sesiones de terminal las gestiona OpenSSH. En paralelo, PQ-SSH puede conectar una sesión libssh para funciones SFTP.

Tareas de flota (ejecutar acciones en muchos servidores)

Las tareas de flota te permiten ejecutar la misma acción en múltiples perfiles guardados en una sola operación. Esto es útil para trabajo rutinario de administración: comprobaciones de salud, estados de servicios y reinicios controlados.

Una acción → muchos destinos

Selecciona 5–50 perfiles y ejecuta la misma tarea en paralelo.

Control de concurrencia

Limita cuántos hosts se ejecutan a la vez (predeterminado: 4) para evitar sobrecarga.

Seguridad por tiempo de espera

Cada destino tiene un timeout de comando para evitar fallos de “colgarse para siempre”.

Rastro de auditoría

Las acciones de flota pueden registrarse en un log de auditoría para rendición de cuentas.

Dónde encontrar Tareas de flota

  • Abrir desde el menú: Herramientas → Tareas de flota…

Conceptos

  • Destino — un perfil/host contra el que se ejecuta la tarea
  • Acción — lo que quieres hacer (Ejecutar comando, Comprobar servicio, Reiniciar servicio)
  • Concurrencia — número máximo de destinos ejecutándose en paralelo
  • Tiempo de espera — cuánto tiempo se permite que un destino se ejecute antes de marcarlo como fallido

Paso a paso: ejecutar una tarea de flota

  1. Abre Herramientas → Tareas de flota…
  2. Selecciona uno o más perfiles destino en la lista.
  3. Elige una Acción:
    • Ejecutar comando — ejecuta un comando de shell en cada host
    • Comprobar servicio — comprueba si un servicio systemd está activo
    • Reiniciar servicio — reinicia un servicio systemd (disruptivo)
  4. Introduce el Comando (o Nombre del servicio) según la acción.
  5. Configura Concurrencia (recomendado: 2–8).
  6. (Si aparece) configura un Tiempo de espera (recomendado: 60–180 segundos).
  7. Haz clic en Ejecutar.
Las tareas de flota crean una conexión SSH separada por destino para evitar reutilización entre hilos y mantener los fallos aislados.

Detalles de acciones

Ejecutar comando

Ejecuta un comando en cada destino y muestra stdout y stderr capturados.

Ejemplos: 
uptime
df -h
uname -a

Comprobar servicio (systemd)

Comprueba si un servicio systemd está activo (no disruptivo).

Ejemplo de nombre de servicio: 
nginx

Reiniciar servicio (systemd)

Reinicia un servicio (disruptivo). PQ-SSH requiere confirmación adicional antes de ejecutar acciones disruptivas.

Advertencia: Reiniciar servicios puede causar interrupciones. Prefiere Comprobar servicio primero.

Tiempos de espera

  • Si un destino excede el timeout, se marca como Fallido con un mensaje de tiempo de espera.
  • Los demás destinos continúan ejecutándose.
  • Los timeouts dejan de esperar salida; no matan de forma fiable procesos remotos.
Consejo: Para tareas largas como apt upgrade, aumenta el timeout o evita Flota.

Concurrencia

Punto de partida recomendado:
  • Casa/lab: 2–4
  • Flota pequeña (10–20 servidores): 4–8

Cancelar una tarea

Haz clic en Cancelar para solicitar cancelación. Los destinos que aún no han comenzado se omiten.

Entender los resultados

  • Estado — Ok / Fallido / Cancelado
  • Duración
  • Salida — recortada para legibilidad

Registro de auditoría (recomendado para Flota)

Las acciones de flota pueden escribirse en un registro de auditoría dedicado para trazabilidad.

Privacidad: Evita almacenar cargas completas de comandos en registros de auditoría si los comandos pueden contener secretos. Prefiere hashes y metadatos mínimos.

Tareas programadas (ejecutar comandos más tarde o según un horario)

Las tareas programadas te permiten definir un comando y una programación, y luego instalarlo en un perfil de servidor seleccionado. Esto es útil para mantenimiento rutinario, acciones diferidas y tareas recurrentes (comprobaciones diarias, reinicios semanales, etc.).

Las tareas programadas se configuran localmente en PQ-SSH, pero solo se ejecutan después de que hagas clic explícitamente en Instalar en el servidor. Crear o editar una tarea no afecta al sistema remoto hasta que se instale.
Las tareas programadas son ideales para tareas diferidas o recurrentes en un solo servidor. Para ejecución inmediata en muchos servidores, usa Herramientas → Tareas de flota….

Dónde encontrar Tareas programadas

  • Abrir desde el menú: Herramientas → Tareas programadas…

Qué ves

La ventana de tareas programadas muestra una lista de tareas y su estado. Cada fila incluye:

  • Nombre — etiqueta amigable para la tarea
  • Perfil — en qué servidor (perfil) se pretende ejecutar
  • Tipo — De una sola vez o Recurrente
  • Cuándo — hora de ejecución (una sola vez) o cadena de programación (recurrente)
  • Comando — el comando que se ejecutará en el servidor
  • Instalada — si la tarea está instalada en el servidor remoto
  • Habilitada — si la tarea está habilitada localmente (uso futuro)

Instalar, Cancelar y Eliminar (qué significa cada acción)

  • Instalar en el servidor — crea o actualiza la entrada del programador en el servidor remoto. No hay cambios remotos antes de este paso.
  • Cancelar en el servidor — elimina la entrada del programador que PQ-SSH instaló previamente en el servidor remoto.
  • Eliminar — elimina la tarea solo de PQ-SSH. No modifica el servidor remoto.
Importante: Eliminar una tarea sin cancelarla primero puede dejarla activa en el servidor remoto.

Tipos de tareas

De una sola vez (ejecutar una vez)

Una tarea de una sola vez se ejecuta una vez a una hora local específica (introducida en el diálogo como YYYY-MM-DD HH:MM). PQ-SSH la instala usando el mejor backend disponible en el servidor.

Recurrente (OnCalendar)

Una tarea recurrente se ejecuta repetidamente según una programación. Cuando systemd está disponible, PQ-SSH usa el formato OnCalendar de systemd (ejemplos: daily, Mon..Fri 02:00). Si los temporizadores de systemd-user no se pueden usar, PQ-SSH recurre a cron.

Selección de backend (automática)

Al instalar una tarea, PQ-SSH sondea el servidor remoto para determinar qué backends de programación están disponibles. Se elige automáticamente el mejor backend disponible.

  • Preferido: temporizadores systemd de usuario (systemctl --user)
  • Fallback recurrente: cron (crontab)
  • Fallback de una sola vez: at (at), o cron si at no está disponible
Algunos servidores tienen systemctl instalado pero no proporcionan una sesión DBus de usuario para inicios de sesión SSH no interactivos. En ese caso, los temporizadores systemd de usuario pueden fallar y PQ-SSH recurrirá a cron/at cuando sea posible.

Entorno de ejecución del comando

Los comandos de tareas programadas se ejecutan de forma no interactiva en el servidor remoto usando: 

/bin/sh -lc 'your command'
No se asigna una terminal interactiva. Los comandos que requieren interacción TTY (como editores o solicitudes de contraseña) fallarán.

Ejemplos

  • Instantánea diaria de uso de disco 
    df -h > ~/disk-usage.txt
  • Reiniciar un servicio por la noche (con cuidado) 
    sudo systemctl restart nginx
  • Comando diferido de una sola vez 
    echo "hello" >> ~/hello.log
Nota de seguridad: Evita incrustar secretos en comandos de tareas programadas. Prefiere leer secretos desde archivos protegidos o gestores de secretos.

Macros (teclas rápidas)

PQ-SSH admite macros de teclas rápidas por perfil. Una macro es un atajo de teclado que envía un comando al terminal SSH activo al instante (opcionalmente añadiendo [Enter] automáticamente).

Dónde viven las macros

  • Abre Editar perfiles…
  • Selecciona un perfil
  • Usa el panel Macros de teclas rápidas a la derecha

Crear una macro

  1. Haz clic en + para añadir una macro.
  2. Configura un Nombre (recomendado).
  3. Haz clic en el campo Atajo y presiona una combinación (p. ej. F2, Alt+X).
  4. Escribe el Comando a enviar.
  5. Habilita Enviar [Enter] automáticamente si quieres que se ejecute de inmediato.
Las macros son intencionadamente simples: PQ-SSH envía texto plano al terminal. No ejecuta nada localmente.

Usar macros

  • Conecta usando el perfil que contiene la macro.
  • Presiona el atajo de la macro mientras la ventana de terminal esté activa.
  • PQ-SSH envía el comando de la macro al terminal.

Marcadores (variables)

Las macros pueden incluir variables de marcador que PQ-SSH expande justo antes de enviar el comando.

Ejemplo: 
echo "Connected to {USER}@{HOST}:{PORT}"
se expande a: 
echo "Connected to john@example.com:2222"

Marcadores admitidos

  • {PROFILE} — nombre del perfil
  • {USER} — usuario SSH
  • {HOST} — hostname/IP
  • {PORT} — puerto (por defecto 22)
  • {TARGET} — user@host (o user@host:port si no es 22)
  • {HOME} — ~
  • {KEYFILE} — ruta del archivo de clave configurado (si existe)
  • {DATE} — fecha actual (YYYY-MM-DD)
  • {TIME} — hora actual (HH:MM:SS)
{KEYFILE} se expande a una cadena vacía si no hay archivo de clave configurado en el perfil.
{TARGET} usa user@host para el puerto 22 y user@host:port para puertos distintos de 22.

Escapar llaves

  • {{ se convierte en una { literal
  • }} se convierte en una } literal
Ejemplo: 
{{{USER}}}
se convierte en: 
{root}
Importante: Los marcadores desconocidos se dejan tal cual.

Ejemplos de macros

  • Estado rápido 
    uptime
  • Mostrar uso de disco 
    df -h
  • Seguir el log de un servicio 
    journalctl -u nginx -f
  • Banner de contexto (con marcadores) 
    echo "=== {PROFILE} :: {TARGET} ==="
  • Comprobar entrada de clave de host 
    ssh-keygen -F {HOST}
Consejo: Prefiere teclas de función (F1…F12) o atajos Alt+… que no entren en conflicto con apps de terminal como vim.

Menú de ajustes

PQ-SSH incluye un diálogo de Ajustes para controlar la apariencia y el comportamiento de registro.

Dónde encontrarlo

  • Desde el menú superior: Archivo → Ajustes…

Qué ocurre al guardar

  • Los ajustes se guardan y se aplican inmediatamente.
  • Los cambios de tema se aplican al instante (no requiere reinicio).
  • Los cambios de nivel de registro se aplican al instante.
Los ajustes se almacenan usando QSettings bajo el nombre de la aplicación y la organización configurados por PQ-SSH.

Temas

Los temas controlan el aspecto de la interfaz principal (listas, botones, diálogos, pestañas). Los colores del terminal se controlan por separado mediante el Esquema de terminal seleccionado en cada perfil.

Temas disponibles

  • CPUNK Dark — interfaz oscura con acentos neón CPUNK
  • CPUNK Orange — interfaz oscura con acentos naranjas
  • Windows Basic — interfaz clara/neutra (sensación clásica de escritorio)
  • CPUNK Neo — tema adicional estilo CPUNK incluido con la app

Cómo cambiar el tema

  1. Abre Archivo → Ajustes…
  2. Selecciona un Tema de la app
  3. Haz clic en Guardar
Si no ves un cambio visible, asegúrate de haber hecho clic en Guardar. PQ-SSH aplica el tema cuando se guardan los ajustes.

Transferencia de archivos

PQ-SSH incluye un flujo de trabajo integrado de transferencia de archivos. Usa una sesión libssh separada para operaciones SFTP, mientras que el terminal interactivo sigue siendo impulsado por OpenSSH.

Pestaña Archivos

La ventana principal incluye una pestaña Archivos junto a la pestaña Log. Cuando SFTP está conectado, puedes navegar por el sistema de archivos remoto y realizar transferencias.

  • Si libssh se conecta correctamente, la pestaña Archivos se activa y el listado remoto funciona.
  • Si libssh falla, la terminal SSH sigue funcionando con normalidad, pero las funciones SFTP se deshabilitan.

Cargas por arrastrar y soltar

  • Si libssh está conectado, los archivos soltados se suben al directorio de trabajo remoto actual ($PWD).
  • Si libssh no está conectado, PQ-SSH guarda el archivo soltado localmente en una carpeta segura de respaldo.

Carpeta local de respaldo

~/pqssh_drops/
Nota: El respaldo existe para evitar “fallo silencioso” cuando el usuario espera que el drop haga algo. No es una carpeta de sincronización.

Razones comunes por las que SFTP está deshabilitado

  • Falla la autenticación de la sesión libssh
  • Problemas de red/firewall
  • El perfil usa un tipo de clave que libssh aún no admite en PQ-SSH
Pestaña Archivos — navega y transfiere archivos vía SFTP (sesión libssh).

Indicador de estado PQ

PQ-SSH ejecuta un sondeo rápido para detectar si el servidor remoto admite un intercambio de claves compatible con PQ. Esto no cambia la configuración del servidor; es informativo.

Qué significan las insignias

Importante: PQ-SSH puede usar dos pilas SSH diferentes al mismo tiempo:
  • SSH de terminal usa tu OpenSSH del sistema (esta es la sesión interactiva principal).
  • SFTP usa una sesión opcional de libssh (solo para la pestaña Archivos / transferencias).
Debido a que estas pilas son diferentes, pueden negociar algoritmos diferentes.

Soporte PQ

Soporte PQ: SÍ Soporte PQ: NO PQ: comprobando…
  • Soporte PQ: SÍ — el sondeo detectó un intercambio de claves PQ/híbrido ofrecido por el servidor.
  • Soporte PQ: NO — el servidor probablemente no ofrece el KEX PQ/híbrido sondeado.
  • PQ: comprobando… — sondeo en progreso.

KEX SSH vs KEX SFTP

PQ-SSH muestra el intercambio de claves negociado por separado para las dos conexiones:

  • KEX SSH — negociado por OpenSSH (la sesión de terminal). Este es el más importante para tu shell interactiva.
  • KEX SFTP — negociado por libssh (la sesión SFTP usada por la pestaña Archivos).
Limitación de libssh: incluso si tu sesión de terminal negocia un KEX PQ/híbrido, la sesión SFTP de libssh puede seguir mostrando algoritmos clásicos por limitaciones de soporte de la biblioteca. Esto es esperado y no “degrada” tu sesión de terminal OpenSSH.

Diagnóstico en la pestaña Log

Con depuración PQ habilitada, puedes ver líneas como: 

[PQ-PROBE] ... ssh ... true
[SFTP-KEX] Classical (libssh limitation) → X25519 (Curve25519)
Consejo: Si KEX SSH muestra PQ/híbrido pero KEX SFTP muestra clásico, normalmente significa: el servidor admite PQ/híbrido en OpenSSH, pero la ruta SFTP de la biblioteca no puede negociarlo.
Los indicadores son pistas UI de “mejor esfuerzo”; PQ-SSH no modifica la configuración del servidor SSH y no impone modo PQ.

Instalar una clave pública en un servidor

PQ-SSH puede instalar una clave pública OpenSSH directamente en el archivo ~/.ssh/authorized_keys del servidor remoto de forma segura y no destructiva.

Esta función está disponible desde:

  • Claves → Instalar clave pública en el servidor… (menú principal)
  • Generador de claves → pestaña Claves → Instalar… (instalar una clave seleccionada)

Qué ves (paso de confirmación)

Antes de cambiar nada en el servidor, PQ-SSH muestra un diálogo de confirmación con:

  • Nombre del perfil destino
  • Usuario / host / puerto
  • Ruta remota: ~/.ssh/authorized_keys
  • Tipo de clave (por ejemplo ssh-ed25519)
  • Vista previa de la clave (un extracto corto de la clave)
No ocurre nada a menos que confirmes explícitamente.

Qué ocurre en el servidor

Después de la confirmación, PQ-SSH se conecta usando SSH/SFTP y realiza estos pasos:

  1. Asegurar que exista el directorio SSH 
    mkdir -p ~/.ssh
chmod 700 ~/.ssh
  2. Leer el authorized_keys existente (si está presente)

    Si la misma clave ya existe, no se cambia nada.

  3. Crear una copia de seguridad con marca de tiempo (si el archivo existe)
    La creación de copia de seguridad es obligatoria. Si falla, la instalación de la clave se aborta.
  4. Añadir la clave solo si falta

    Nunca se insertan claves duplicadas.

  5. Corregir permisos 
    chmod 600 ~/.ssh/authorized_keys

Garantías de seguridad

  • Se conservan las claves existentes
  • Copia de seguridad antes de modificar
  • Sin claves duplicadas
  • Se fuerzan permisos SSH correctos
  • No se suben contraseñas ni claves privadas

Registro & depuración

Niveles de registro

La verbosidad del registro se puede controlar desde Archivo → Ajustes…:

  • Solo errores — registros mínimos
  • Normal — valor predeterminado recomendado
  • Depuración — registros detallados para solución de problemas

Abrir el archivo de registro

Usa el menú: Ver → Abrir archivo de registro

~/.local/share/pq-ssh/logs/pq-ssh.log

Modo de depuración PQ (UI)

Habilita depuración PQ en la barra inferior para mostrar diagnósticos extra (detalles del comando SSH, listas de esquemas, mensajes de bajo nivel).

Nota de privacidad: PQ-SSH no registra contraseñas ni contenido de claves privadas. La salida de depuración puede contener nombres de host y rutas de archivo.
Registro de auditoría — revisa acciones (especialmente tareas de flota) para trazabilidad.

Claves & advertencias de caducidad

Abre Claves → Generador de claves… para crear/gestionar claves.

  • Al iniciar, PQ-SSH puede marcar automáticamente claves caducadas en los metadatos.
  • Si existen claves caducadas, se muestra una advertencia en el área de estado.
  • Rota las claves regularmente para mejor seguridad.

Solución de problemas

La instalación falló: “Failed to connect to bus” / “No medium found”
  • Esto indica que systemctl --user está presente, pero no hay una sesión DBus de usuario disponible en el servidor.
  • Común en servidores headless o mínimos sin “user lingering” habilitado.
  • Arreglar una vez en el servidor (recomendado): 
    sudo loginctl enable-linger <user>
  • Alternativamente, PQ-SSH recurrirá a cron o at si están disponibles.
La tarea de flota falló en un host
  • Abre Ver → Abrir archivo de registro para errores detallados.
  • Intenta ejecutar la misma tarea contra ese host único para reproducir.
  • Reduce Concurrencia si muchos hosts fallan a la vez.
  • Aumenta Tiempo de espera si el comando es lento en ese host.
El reinicio de flota es peligroso — ¿cómo hacerlo de forma segura?
  • Ejecuta Comprobar servicio primero para entender el estado actual.
  • Reinicia primero un host, verifica el comportamiento y luego amplía al grupo.
  • Evita reinicios durante upgrades del SO o instalaciones de paquetes.
  • Usa una concurrencia conservadora (2–4) para flotas en producción.
El cambio de tema no se aplicó
  • Abre Archivo → Ajustes…, selecciona un tema y haz clic en Guardar.
  • Si es necesario, cierra y vuelve a abrir el diálogo de Ajustes e inténtalo de nuevo.
La conexión falla inmediatamente
  • Verifica host/IP y puerto en el perfil.
  • Revisa el archivo de registro (Ayuda → Abrir archivo de registro).
  • Habilita depuración PQ para diagnósticos SSH detallados.
PQ muestra OFF
  • El servidor probablemente no ofrece el intercambio de claves PQ sondeado.
  • Esto es informativo; el SSH normal aún puede ser seguro según tus políticas.
La pestaña Archivos / SFTP está deshabilitada
  • La sesión libssh puede fallar aunque el SSH de terminal funcione.
  • Algunos tipos de claves aún no son compatibles con libssh.
  • Revisa los registros para errores de autenticación o conexión.
Arrastrar y soltar se guardó localmente en lugar de subir
  • Esto significa que SFTP/libssh no estaba conectado en el momento del drop.
  • Comprueba si SFTP está listo e inténtalo de nuevo, o usa transferencias desde la pestaña Archivos.
  • Carpeta local de respaldo: ~/pqssh_drops/
El atajo de la macro no se activa
  • Asegúrate de que la macro tenga ambos: Atajo y Comando.
  • Asegúrate de que la ventana de terminal sea la ventana activa.
  • Evita atajos usados por apps de terminal (p. ej. algunos Ctrl+… en vim).
  • Prueba una tecla de función como F2 o un atajo Alt+….
Los marcadores no se expandieron
  • Asegúrate de usar tokens soportados como {HOST}, {USER}, {PORT}.
  • Los marcadores desconocidos se dejan sin cambios.
Demasiada salida de depuración en la UI
  • Desactiva depuración PQ.
  • La app ocultará líneas ruidosas como wrappers del comando SSH por defecto.
